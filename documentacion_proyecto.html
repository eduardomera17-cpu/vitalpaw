Filosofía e Arquitectura "Offline-First"

El sistema móvil de VitalPaw PetShop ha sido diseñado bajo los estándares de ingeniería más estrictos para garantizar operatividad y resiliencia en escenarios inestables (zonas sin señal celular, sótanos o áreas rurales). La aplicación adopta el patrón arquitectónico Offline-First: el teléfono móvil nunca consume recursos externos de forma síncrona; todas las operaciones críticas de lectura y escritura se realizan contra la base de datos interna relacional de alta eficiencia SQLite, orquestada mediante la librería de persistencia tipada Drift (Dart).

Cuando hay conectividad a Internet, un robusto motor de sincronización asíncrono (el SyncWorker) despierta de forma pasiva ante cambios de red informados por el ConnectivityMonitor y gestiona la transferencia bidireccional de datos con Google Firebase Firestore y Firebase Auth.

Estructura de Sincronización FIFO Jerárquica

Para evitar que se generen registros huérfanos en la nube y violar la integridad referencial de la base de datos distribuida, el motor sigue un estricto orden jerárquico unidireccional de dependencias al procesar colas pendientes:

¿Por qué este orden? Las reglas de seguridad de Firestore impiden subir una mascota si su dueño (propietario) aún no existe. Igualmente, no se puede agendar una cita o un pedido sin una mascota o usuario válidos previamente indexados en el servidor en la nube.

Estrategias de Resolución de Conflictos

Debido a la naturaleza asíncrona del sistema distribuido offline, la aplicación implementa tres robustas estrategias de conflicto dependiendo del valor crítico de la información:

Arquitectura Limpia y Desglose de Carpetas Clave

El proyecto móvil de VitalPaw PetShop adopta la arquitectura de software recomendada Clean Architecture, aislando completamente las reglas de negocio de los detalles de infraestructura. Esto permite realizar pruebas unitarias robustas e independientes y flexibilizar el cambio de servicios de backend.

Core Layer lib/core

Configuraciones y utilidades compartidas del sistema. Contiene el despachador de navegación global (app_router.dart) impulsado por GoRouter con guardias de seguridad de sesión, y los proveedores globales de Drift SQLite (database_provider.dart).

Domain Layer lib/domain

El corazón puro del negocio. Contiene las Entidades (entities/) inmutables que representan las estructuras de datos (citas, pedidos, registros de mascotas), interfaces abstractas para repositorios (repositories/) y los Casos de Uso (usecases/) puros libres de cualquier dependencia de frameworks (Drift/Firebase).

Data Layer lib/data

La infraestructura física del sistema. Contiene los orígenes de datos remotos en la nube (datasources/remote) con llamadas directas a Firebase, y locales (local/database) con la estructuración de tablas relacionales en Drift SQLite. También implementa los repositorios del dominio, estrategias de conflictos de sincronización y servicios compartidos (GPS, pasarela de pago).

Presentation Layer lib/presentation

La capa visual y de interacción del usuario. Contiene las pantallas (screens/), los widgets pequeños reutilizables (widgets/) y los proveedores de estado reactivo (providers/) de Riverpod que conectan de forma síncrona los casos de uso del dominio con la UI.

AppDatabase

lib/data/local/database/app_database.dart

Central de la base de datos local SQLite del celular, gestionada con Drift. Agrupa las 8 tablas de la app, expone los DAOs correspondientes, activa las llaves foráneas para integridad física (foreign_keys = ON) y optimiza el almacenamiento al forzar lecturas y escrituras concurrentes de alta velocidad mediante el modo WAL (journal_mode = WAL). 

@DriftDatabase(
  tables: [UsersTable, PetsTable, AppointmentsTable, MedicalRecordsTable, ProductsTable, OrdersTable, OrderItemsTable, LocationTracksTable],
  daos: [UsersDao, PetsDao, AppointmentsDao, MedicalRecordsDao, ProductsDao, OrdersDao, LocationTracksDao],
)
class AppDatabase extends _$AppDatabase {
  AppDatabase() : super(_openConnection());

  @override
  int get schemaVersion => 1;

  @override
  MigrationStrategy get migration => MigrationStrategy(
    onCreate: (m) async {
      await m.createAll(); // Crea todas las tablas vacías automáticamente en la instalación
    },
    beforeOpen: (details) async {
      // 1. Activamos la Integridad Referencial. Esto evita registros huérfanos localmente
      await customStatement('PRAGMA foreign_keys = ON');

      // 2. Activamos WAL (Write-Ahead Logging) para escrituras concurrentes rápidas sin bloquear la pantalla
      await customStatement('PRAGMA journal_mode = WAL');
    },
  );
}

Stream<List<AppointmentsTableData>> watchByDate(DateTime date) {

  final end = start.add(const Duration(days: 1));

  return (select(db.appointmentsTable)
        ..where((t) =>
            t.scheduledAt.isBiggerOrEqualValue(start) &
            t.scheduledAt.isSmallerThanValue(end) &
            t.deletedAt.isNull(),

        ..orderBy([(t) => OrderingTerm.asc(t.scheduledAt)]))
      .watch();

Motor de Sincronización

ConnectivityMonitor

lib/data/sync/connectivity_monitor.dart

Funciona como un sensor pasivo de Internet para la app celular. Monitorea cambios físicos de conectividad inalámbrica (WiFi o datos celulares) y los reduce a dos estados limpios: online y offline, los cuales transmite a toda la app mediante un Stream broadcast de Riverpod. 

class ConnectivityMonitor {
  final Connectivity _connectivity;
  final _controller = StreamController<NetworkStatus>.broadcast();

  void _init() {
    // Escucha el estado físico real de la red celular y emite alertas reactivas
    _connectivity.onConnectivityChanged.listen((results) {
      final status = results.any((r) => r != ConnectivityResult.none)
          ? NetworkStatus.online
          : NetworkStatus.offline;
      _controller.add(status);
    });
  }
}
Estrategia de Conflicto

LwwStrategy

lib/data/sync/strategies/lww_strategy.dart

Resuelve choques de datos usando la estampa de tiempo de actualización updatedAt (El último cambio gana). Se aplica automáticamente a entidades donde el último valor modificado representa el estado real deseado, como el perfil de un usuario, el stock en inventario de productos o la confirmación de una cita. 

/// Compara fechas y decide quién tiene el dato correcto (El celular o la Nube).
Future<ConflictResult> resolve({
  required String collection,
  required String docId,
  required DateTime localUpdatedAt,
  required int localSyncVersion,
  required Map<String, dynamic> localData,
}) async {
  final remoteDoc = await _firestore.collection(collection).doc(docId).get();
  if (!remoteDoc.exists) return ConflictResult.localWins; // Registro nuevo local

  final remoteData = remoteDoc.data()!;
  final remoteUpdatedAt = (remoteData['updatedAt'] as Timestamp).toDate();

  // El cambio más reciente (local vs nube) se consagra como ganador
  return localUpdatedAt.isAfter(remoteUpdatedAt)
      ? ConflictResult.localWins
      : ConflictResult.remoteWins;
}
Estrategia de Conflicto

MergeStrategy

lib/data/sync/strategies/merge_strategy.dart

Garantiza la seguridad médica de las mascotas. Impide colisiones de recetas y diagnósticos que se modifiquen en paralelo. En lugar de sobreescribir la información sensible, devuelve un estado de conflicto clínico para forzar la intervención humana por parte del veterinario en turno. 

/// Evalúa si el historial médico chocó en versiones distribuidas
Future<MergeResult> evaluate({
  required String petId,
  required String recordId,
  required int localSyncVersion,
  required Map<String, dynamic> localData,
}) async {
  final remoteDoc = await _firestore.collection('pets').doc(petId)
      .collection('medical_records').doc(recordId).get();

  if (!remoteDoc.exists) return MergeResult.safe; // Seguro subir (nota nueva)

  final remoteData = remoteDoc.data()!;
  final remoteSyncVersion = (remoteData['syncVersion'] as num?)?.toInt() ?? 0;

  // Si las versiones de sincronización difieren, ¡HAY CONFLICTO!
  if (localSyncVersion != remoteSyncVersion) return MergeResult.conflict;
  return MergeResult.noChange;
}
Estrategia de Conflicto

AppendStrategy

lib/data/sync/strategies/append_strategy.dart

Optimización logística de alta frecuencia. Acumula en cola local las coordenadas GPS del chofer recolectadas offline y las procesa en lotes de escritura atómicos en la nube de máximo 400 elementos, auto-depurando datos locales históricos superiores a 7 días. 

/// Sube en lote todas las ubicaciones GPS del repartidor acumuladas offline
Future<void> uploadPendingTracks(String driverId) async {
  final pending = await _tracksDao.getPendingByDriver(driverId);
  if (pending.isEmpty) return;

  final byOrder = <String, List<LocationTracksTableData>>{};
  for (var t in pending) { byOrder.putIfAbsent(t.orderId, () => []).add(t); }

  for (var entry in byOrder.entries) {
    // Dividimos las coordenadas en paquetes de 400 para respetar el límite de 500 de Firestore
    final batches = _chunked(entry.value, 400);
    for (var chunk in batches) {
      final batch = _firestore.batch();
      for (var track in chunk) {
        final ref = _firestore.collection('orders').doc(entry.key).collection('tracking').doc(track.id);
        batch.set(ref, _toMap(track));
      }
      await batch.commit(); // Commitea el lote completo de golpe en red
      for (var track in chunk) { await _tracksDao.markSynced(track.id); }
    }
  }
  // Eliminación física del teléfono de puntos GPS viejos para no saturar memoria celular
  await _tracksDao.deleteOldSynced(keepDays: 7);
}
Motor de Sincronización

SyncDispatcher

lib/data/sync/sync_dispatcher.dart

El cerebro motor del Offline-First. Ejecuta la sincronización secuencial en el orden estricto de dependencias y maneja una lógica resiliente de reintentos con Backoff Exponencial y Jitter para evitar la saturación en picos de red. 

/// Sistema de reintentos inteligente con Backoff Exponencial y Jitter
Future<void> _uploadWithBackoff(Future<void> Function() operation, {int maxRetries = 5}) async {
  int attempt = 0;
  while (true) {
    try {
      await operation();
      return; // Éxito absoluto en la red
    } catch (e) {
      attempt++;
      if (attempt >= maxRetries) rethrow;

      // Fórmula: 2^attempt con desfasamiento aleatorio (Jitter) para no colapsar el servidor
      final baseDelay = pow(2, attempt).toInt();
      final jitterMs = Random().nextInt(500);

      await Future.delayed(Duration(seconds: baseDelay, milliseconds: jitterMs));
    }
  }
}
Motor de Sincronización

SyncWorker

lib/data/sync/sync_worker.dart

Hilo/Asistente de segundo plano reactivo. Se suscribe a las alertas de ConnectivityMonitor. Si no hay señal se suspende en reposo, pero ante la recuperación inmediata de red despierta al despachador principal para realizar la subida masiva. 

class SyncWorker {
  final SyncDispatcher _dispatcher;
  final ConnectivityMonitor _connectivity;
  SyncState _state = SyncState.idle;

  void start({String? driverId}) {
    _currentDriverId = driverId;

    // Sincronización inmediata al arrancar si ya gozamos de red estable
    _connectivity.currentStatus.then((status) {
      if (status == NetworkStatus.online) _runSync();
    });

    // Escucha reactiva ante reconexiones de señal
    _subscription = _connectivity.statusStream.listen((status) {
      if (status == NetworkStatus.online && _state == SyncState.idle) {
        _runSync();
      }
    });
  }
}

Flujos de Control Clave del Ecosistema

Comprende los cuatro flujos lógicos críticos del sistema explicados paso a paso mediante trazas y dependencias internas de control.

1. Flujo de Sincronización Automática (Sync Engine)

Detección de Conexión Física
El hardware móvil reconecta a red móvil o WiFi. ConnectivityMonitor captura el cambio físico a nivel de sistema operativo y emite un estado online reactivo.
Activación del Trabajador de Fondo
El SyncWorker capta la señal del monitor, sale del modo de reposo (SyncState.idle) y actualiza su estado en pantalla a SyncState.syncing activando la rotación de nubes.
Filtro de Colas Pendientes y Orden Jerárquico
SyncDispatcher lee secuencialmente de Drift SQLite las colas pendientes (syncStatus = 'pending') de cada una de las 8 tablas en estricto orden FIFO.
Evaluación de Conflictos y Subida de Red
El despachador valida las versiones distribuidas con LwwStrategy o MergeStrategy. Si el cambio local es correcto, ejecuta la subida llamando a Firestore protegida por reintentos de Backoff Exponencial y Jitter.
Sello de Sincronización Local
Al recibir confirmación del backend, el DAO local actualiza la base SQLite modificando el syncStatus a synced, refrescando visualmente las interfaces.

2. Gestión de Conflictos Clínicos y Merge Asistido

Choque de Versiones
Durante la sincronización de una nota médica, el MergeStrategy detecta que el syncVersion local no coincide con la versión guardada en la nube por otro veterinario.
Bloqueo y Rotulación en SQLite
El motor frena la subida de inmediato para evitar sobreescribir datos de salud. Marca localmente el registro con MedicalSyncStatus.conflict y anota los detalles del choque en la base Drift.
Alerta Visual en AppBar
El SyncStatusBadge visualiza una nube naranja cruzada de advertencia. El Riverpod Provider actualiza y notifica conflictos activos en la clínica veterinaria.
Fusión Manual en Pantalla
El veterinario abre ConflictResolverScreen, revisa las discrepancias, redacta las notas oficiales de unificación clínica en el formulario _ConflictResolutionForm y aprueba la resolución.
Unificación y Reenvío de Datos
Se ejecuta resolveConflictUseCaseProvider, el cual graba la nota reconciliada, eleva la versión en SQLite a synced y sube a Firestore la resolución con la aprobación del doctor.

3. Rastreo GPS de Reparto y Optimización "Append-Only"

Inicio del Servicio Logístico
El chofer marca en pantalla un pedido como "En camino". GpsService solicita acceso fino e inicia la recopilación de posiciones geográficas persistentes, bloqueando suspensiones con WakeLock.
Colección Offline y Escritura en SQLite
Cada 5 segundos, la app captura la latitud, longitud y rumbo. Graba cada registro en la tabla local LocationTracksTable mediante Drift, evitando conexiones móviles instantáneas e innecesarias.
Subida en Paquetes (WriteBatch)
Al recuperar señal estable de red, el TrackingUploader y AppendStrategy agrupan las trazas por orden de pedido, las empaquetan en arrays máximos de 400 puntos y las suben a Firestore de un solo commit atómico.
Posicionamiento Rápido de Bajo Coste
Paralelamente, el uploader actualiza los campos planos latestLat y latestLng en la raíz del documento del pedido para permitir el trackeo barato en vivo del cliente sin consultas pesadas.
Auto-Limpieza de Memoria Local
Tras la sincronización, el despachador ejecuta deleteOldSynced(keepDays: 7) sobre el DAO local, liberando permanentemente espacio físico de almacenamiento del celular.

4. Ciclo Completo de Citas y Recordatorios de WhatsApp

Formulación y Agenda
El cliente o administrador reserva un turno desde AppointmentFormScreen validando la disponibilidad y seleccionando un horario.
Registro SQLite
El DAO ejecuta upsertAppointment, asegurando el bloqueo inmediato del turno en la base de datos interna SQLite y previniendo la duplicación accidental offline.
Propagación Firestore
El motor asíncrono sube los detalles a la colección centralizada en la nube de Google Firestore, quedando inmediatamente disponible para todo el personal.
Disparo de Alerta de WhatsApp
El sistema solicita enviar el recordatorio. Se llama a WhatsappReminderService.sendReminder, que toma la cita, limpia caracteres raros del teléfono y redacta el mensaje de bienvenida.
Redirección Nativa de Mensajería
Se lanza la URL Scheme compatible wa.me a nivel nativo abriendo directamente el canal de chat del cliente con su recordatorio pre-llenado de confirmación.
